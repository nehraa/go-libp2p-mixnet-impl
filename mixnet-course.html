01

What is Mixnet?

The Privacy Problem

When you send data over the internet, anyone watching the network can see who you're talking to, when, and how much. This is called metadata leakage.

Imagine sending a letter through the postal system, but every envelope had your return address, the recipient's address, and a timestamp visible on the outside. Even if the letter's contents are sealed in an envelope, the outside information reveals a lot.

Why should you care? Metadata can reveal your communication patterns, social networks, and even your location — even when the actual message content is encrypted. For journalists, activists, or anyone sensitive to surveillance, this is a serious problem.

How Mixnet Solves It

Lib-Mix is a mixnet implementation for libp2p that adds privacy by:

Route through multiple relays
Your data doesn't go directly to the destination. It bounces through several relay nodes, each only knowing the previous and next hop.
Layer encryption (onion routing)
Like peeling an onion, each relay peels off one encryption layer to learn only where to forward next — never knowing the original sender.
Payload sharding (optional)
Large messages can be split into multiple shards that travel over different paths, making it even harder to correlate traffic.
Padding and timing obfuscation
Random padding hides the actual message size, and timing variations prevent traffic analysis.

02

Meet the Actors

The Cast of Characters

Every software system has components that work together. In Mixnet, there are four main actors:

Origin (You)
The sender. You create data, encrypt it in layers, and send it into the mixnet. Located in upgrader.go — the main Mixnet type.
Relays
Intermediate nodes that forward your encrypted data. They don't know who sent it or where it ultimately goes. Implemented in the relay/ package.
Destination
The final recipient. They receive the fully decrypted message and deliver it to the application.
Circuit Manager
The organizer. It builds and manages the relay paths (circuits) that data travels through. Located in circuit/manager.go.

How They Talk to Each Other

Y
I want to send a message to the destination. First, I need to find some relays to route through.
C
Circuit Manager here! I'll use the relay discovery system to find available relays and build a circuit.
1
Hey, I'm Relay 1! The circuit manager connected me to the origin. I'll forward whatever comes my way to the next relay.
2
I'm Relay 2. I don't know who sent this or where it's going. I just decrypt one layer, find the next hop, and pass it on.
D
Destination here! I've decrypted all layers and can now deliver the original message to the recipient app.
Zero-knowledge forwarding: This is the key property. Relays perform zero-knowledge forwarding — they forward data without knowing the original sender or final destination. Each relay only knows its immediate neighbors.

03

How Data Travels

The Journey of a Message

Let us trace what actually happens when you send a message through Mixnet:

1. Discover Relays Find available relay nodes using libp2p routing
2. Build Circuit Create path: Origin - Relay1 - Relay2 - Relay3 - Destination
3. Encrypt Layers Wrap message in onion encryption: each layer has next-hop info
4. Add Privacy Apply padding (hides size), optionally shard (splits message)
5. Send Through Relays Each relay peels one layer, learns next hop, forwards remaining
6. Destination Reconstructs Final relay peels last layer, delivers original message

Looking at Real Code: The Onion Header

This is how Mixnet builds the layered encryption header. Each layer contains instructions for the next relay:

CODE 
// Build layers from destination back to origin
current := buildHopPayload(isFinal, nextHop, current)

// Each relay gets: next_hop + encrypted_payload
for i := len(c.Peers) - 1; i >= 0; i-- {
  isFinal := i == len(c.Peers)-1
  nextHop := ""
  if !isFinal {
    nextHop = c.Peers[i+1].String()
  }
  plain, err := buildHopPayload(isFinal, nextHop, current)
  enc, err := encryptHopPayload(hopKeys[i], plain)
  current = enc  // Wrap this layer
}
PLAIN ENGLISH

The code builds onion layers BACKWARDS, starting from the destination and wrapping toward the origin.

For the last relay (before destination), "isFinal" is true because it is the final hop.

Each layer gets the next hop address. Relay 2 knows to send to Relay 3, etc.

"buildHopPayload" creates the instructions: "forward this to next address".

Then "encryptHopPayload" wraps those instructions in encryption using the relay key.

This loops backward through all relays, creating layers like wrapping paper.

Why backwards? Starting from the destination and working backward ensures each relay can peel its layer and find the correct next hop. It is like writing instructions for each person in a chain by starting with the last person.

The Privacy Packet Format

Mixnet uses a specific packet format to carry routing information:

CODE 
// PrivacyShardHeader - what goes in each packet
type PrivacyShardHeader struct {
  SessionID   []byte  // ID to reassemble shards
  ShardIndex uint32  // Which shard is this?
  TotalShards uint32  // How many total?
  HasKeys     bool    // Keys included?
  KeyData     []byte  // Encrypted key material
  AuthTag     []byte  // Authenticity proof
  Padding     []byte  // Random bytes for size hiding
}
PLAIN ENGLISH

SessionID: A tracking number that lets the destination know which shards belong together.

ShardIndex: "I am the 2nd piece of 5 total pieces" - helps reconstruct in order.

TotalShards: "There are 5 pieces total" - knows when reconstruction is complete.

HasKeys: Is this the first shard that carries encryption keys?

KeyData: The decryption keys (encrypted so relays cannot read them).

AuthTag: A cryptographic proof that this shard was not tampered with.

Padding: Random garbage data to hide the real message size from observers.

Circuit Construction

What is a Circuit?

A circuit is a pre-established path through relays. Think of it like booking a flight with connections - you know the entire route in advance: Origin - Relay1 - Relay2 - Relay3 - Destination.

type Circuit struct {
    ID      string    // Unique identifier for this circuit
    Peers   []peer.ID // Ordered relays: [Relay1, Relay2, Relay3]
    State   CircuitState // pending - building - active - failed/closed
    CreatedAt time.Time
    UpdatedAt time.Time
}
Why pre-build circuits? Building circuits takes time (discovering relays, establishing connections). By building them once and reusing them for multiple messages, Mixnet saves significant latency.

Circuit States

Circuits have a lifecycle - they go from created to active to eventually closed:

?
Pending Allocated but not built
-
B
Building Establishing connections
-
A
Active Ready for traffic
-
X
Failed/Closed No longer usable
Circuit Manager
The CircuitManager (in circuit/manager.go) handles building circuits, checking their health via heartbeats, and recovering from failures by building new circuits.

Session Routing: An Optimization

Mixnet has an optional optimization called session routing that reduces overhead for repeated messages:

Without session routing: Every message carries all the routing and setup information needed - like bringing your passport for every single flight.
With session routing (EnableSessionRouting=true): The first message on a circuit sends a setup frame with full routing info. Subsequent messages just send a smaller data frame that reuses the cached route.
Security trade-off: Session routing is faster but introduces a subtle risk. A malicious relay can potentially inject data for a known session since relays cache the route state. The destination still verifies end-to-end encryption.

Privacy Features

Padding: Hiding Message Size

If an observer sees you send exactly 1,337 bytes every 5 minutes, they can correlate that with your encrypted messages. Padding adds random bytes to hide the real size:

CODE 
// PrivacyPaddingConfig
type PrivacyPaddingConfig struct {
  Enabled  bool   // Turn padding on/off
  MinBytes int   // Minimum padding (default 16)
  MaxBytes int   // Maximum padding (default 256)
}
PLAIN ENGLISH

Enabled: Whether to use padding at all.

MinBytes: Add at least this many random bytes (default 16).

MaxBytes: Add at most this many random bytes (default 256).

So a 100-byte message might become 116-356 bytes. Observers cannot tell the original size.

Sharding: Splitting for Defense in Depth

For extra privacy, Mixnet can split a large message into multiple shards that travel over different circuits:

Y
I have a 1MB file to send. Instead of one big packet, I will split it into 5 shards of 200KB each.
C
Circuit Manager: I will build 5 different circuits through different relay combinations!
1
Relay 1: I only see Shard 1 going to Circuit A. I have no idea about Shards 2-5.
D
Destination: I receive all 5 shards and use the SessionID to reassemble them into the original file.
Why does this help? An observer would need to correlate traffic across multiple circuits to even know a single message was sent. It is much harder than watching one path.

Auth Tags: Proving Integrity

How do you know the data was not tampered with mid-transit? Auth tags:

XChaCha20-Poly1305
Mixnet uses XChaCha20-Poly1305 for authenticated encryption. This provides both confidentiality (nobody can read it) and authenticity (nobody can modify it without detection).
The auth tag is like a wax seal Just like a wax seal on an envelope proves it was not opened, an auth tag proves the message was not modified. If anyone tries to alter the encrypted data, the auth tag verification fails at the destination.

The Big Picture

Putting It All Together

Let us see how all the pieces fit together in a complete message flow:

Origin
1) Call NewMixnet() - validates config, initializes subsystems
Origin
2) Call OpenStream or Send - discovers relays, builds circuit
Origin
3) Payload: compress - encrypt - pad - (optionally) shard
Relay
4) Each relay: decrypt ONE layer - extract next hop - forward payload
Dest
5) Final relay delivers to destination - reconstruct (if sharded) - deliver via stream

Configuration: What Can You Tune?

MixnetConfig {
  // Circuit setup
  HopCount: 3,          // How many relays in each circuit
  CircuitCount: 3,      // How many parallel circuits

  // Encryption
  EncryptionMode: "full" | "header-only",

  // Privacy features
  EnableSessionRouting: false,
  SessionRouteIdleTimeout: 30s,

  // Padding
  Padding: { Enabled: true, MinBytes: 16, MaxBytes: 256 },

  // Selection
  SelectionMode: "rtt" | "random" | "hybrid",

  // Limits
  MaxBandwidth: 100MB,
  MaxCircuits: 100,
}
